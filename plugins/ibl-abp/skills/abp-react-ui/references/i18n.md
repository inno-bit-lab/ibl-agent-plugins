# i18n keys, fetch flow, fallback labels

> **Sibling references:** `shared-components.md` for the UI primitives,
> `design-tokens.md` for visual tokens, `data-fetching.md` for the
> end-to-end Backend→UI hand-off.

## How localization gets to the React app

The React app does NOT bundle the project's translations. They live in
the backend at `Localization/{ResourceName}/{lang}.json` and are served
via `GET /api/abp/application-localization?CultureName={lang}`.

On the React side:

1. After OIDC login (and on page reloads), `fetchAppConfig(token)` runs in
   `AuthContext.tsx` and pulls the user's culture-specific bundle.
2. The bundle is fed into i18next via `addResourceBundle(culture, 'translation', ...)`.
3. `useTranslation()` reads from i18next; `t('Permission:Foo')` returns
   the localized string.

This means:

- New keys you add to backend `*.json` files are visible after a backend
  rebuild + a React hard refresh (or a logout/login).
- Cache: i18next keeps the bundle in memory for the session. A
  newly-added key without a refresh shows the **raw key**
  (e.g. `Menu:Customers`) — a recurring confusion. Always tell the user
  to hard refresh after adding keys.

## Keys you must add for a new CRUD page

Add to **every** language file in the project (typically `it.json`,
`en.json`, `fr.json`, `es.json`). Missing translations silently fall
through to the raw key on that locale.

| Key | Used by |
|---|---|
| `Menu:{Plural}` | Sidebar entry (or `fallbackName` if missing) |
| `{Plural}` | Page title and CardTitle |
| `{Entity}` | Singular label (used in dialog header, error contexts) |
| `New{Entity}` | "New" button at the top right of the page |
| `{FieldName}` (one per visible column / form field) | Table headers, form labels |
| `Permission:{Plural}` | Permission management UI in admin console |
| `Permission:{Plural}.Create` | ditto |
| `Permission:{Plural}.Edit` | ditto |
| `Permission:{Plural}.Delete` | ditto |
| `{Entity}DeletionConfirmationMessage` | Confirm dialog when deleting |
| `Enum:{EnumName}.{ValueName}` (one per enum value) | Enum filters, table cells, select options |
| `{RootNs}:{Plural}:{ErrorCode}` (one per lifecycle BusinessException code) | Toast on lifecycle errors |

## Interpolation — DOUBLE braces

react-i18next interpolates `{{name}}` (double braces). Single braces
`{name}` are NOT interpolated and will render literally. Make sure every
placeholder in your `.json` files uses double braces:

```jsonc
{
  "Customers:Header.Count": "{{count}} clienti totali nel workspace",
  "Customers:DeleteTitle":  "Elimina cliente \"{{name}}\"?",
  "Pagination:Range":       "{{from}}–{{to}} di {{total}}",
  "Search:MinChars":        "Digita almeno {{n}} caratteri per cercare"
}
```

In the JSX:

```tsx
t('Customers:Header.Count', { count: totalCount })
t('Customers:DeleteTitle', { name: customer.legalName })
t('Search:MinChars', { n: SEARCH_MIN_CHARS })
```

ABP backend exception messages (`Ibl360:Customers:InvalidStatusTransition`
etc.) use SINGLE braces `{from}` and `{to}` because they're interpolated
by the BACKEND, not by react-i18next. Don't touch those.

## Calling `t()` — three patterns

```tsx
// 1. Plain key. Best for stable, project-owned strings.
t('LegalName')

// 2. Key with fallback. Use for keys that might not exist in every
//    project locale, or for "I'm writing the key now, the JSON catches
//    up later" situations. The fallback is shown verbatim when the
//    key is missing.
t('::All', 'All')
t('Customers')   // no fallback — show '{Customers}' raw if missing

// 3. ABP-style resource prefix. The shared ABP packages put their keys
//    under named resources separated by '::'. Use as-is:
t('AbpUi::SavedSuccessfully')
t('AbpIdentity::Users')
t('AbpAccount::Save')

// 4. Dynamic interpolation from a runtime value. Combine with name-based
//    enum keys for dependable lookup:
t(`Enum:CustomerStatus.${row.status}`)
```

## Fallback names in routing

Sidebar entries need a `fallbackName` so that during the brief moment
between app boot and bundle load, users see "Customers" not
"Menu:Customers". See `routing-menu.md`.

The Sidebar component already supports this; just add the field on every
new `route-config.ts` entry.

## When the user reports "the menu shows the raw key"

Most common causes, in order:

1. **Backend was down** when the React app loaded — bundle fetch failed
   silently. Fix: restart backend, hard refresh.
2. **i18next bundle is cached** in memory from a prior session before the
   key existed. Fix: hard refresh (Ctrl+F5) or logout/login.
3. **Backend has the key but in the wrong resource file** — the project
   uses `Ibl360` resource, but the key was added to `AbpUi` by mistake.
   Verify via:

   ```bash
   curl -k -s 'https://localhost:PORT/api/abp/application-localization?CultureName=en&OnlyDynamics=false' \
     | python -c "import sys,json; r=json.load(sys.stdin)['resources']['Ibl360']['texts']; print(r.get('Menu:Customers','MISSING'))"
   ```

4. **The key was added but the backend wasn't restarted in production
   mode** — dev mode hot-reloads `.json` via `ReplaceEmbeddedByPhysical`
   in `*Module.cs`. Prod mode loads embedded resources at startup only.

## When you should ALSO add keys locally in `react/src/locales/`

Almost never. The project uses backend-served localization for
consistency. Local React bundles are only useful when:

- Pre-login (no auth token, so no backend bundle yet).
- Strings that are React-only and not shared with other clients.

If you do add local keys, register them via `i18n.addResourceBundle()`
in `i18n/i18n.ts` and DON'T duplicate keys that the backend serves —
backend wins.
